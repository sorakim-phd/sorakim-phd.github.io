🧩 Background

invisible delays in feedback loops

In GMP environments, document review cycles between QC and QA are critical yet often opaque. Each deviation or CAPA record passes through multiple reviewers, but visibility into review ownership, timing, and bottleneck formation is limited.

As review loops lengthen, communication often becomes anecdotal — “still under review,” “waiting for QA,” or “someone has it” — rather than data-based. This creates frustration, slows coordination, and makes audit-facing workflow management more stressful.

The core problem is not only delay itself, but the lack of a shared operational view of delay.

⚠️ Observation

workflow bottlenecks without visibility

Observed pain points

  • Teams often knew a document was delayed, but not where the delay was occurring.
  • Some reviewers accumulated more pending items than others, but the imbalance was hard to see in real time.
  • Workflow judgments were made through verbal updates rather than shared performance signals.
  • Latency patterns were often recognized only after timelines were already affected.

Why this mattered

These were not simply communication issues. They reflected a missing observability layer in the documentation workflow — one that could make hidden review work visible before delays cascaded further.

💬 In regulated review systems, delay is not only a timing problem — it is also a visibility problem.

🎯 Design question

HCI framing

How can feedback flow, reviewer bottlenecks, and workload distribution be visualized without modifying validated documentation systems?

🧠 Method

metadata structuring + workflow analytics

1. Data extraction

Collected metadata from deviation and CAPA records, including creation date, submission date, feedback date, revision date, closure date, reviewer ID, analyst ID, revision count, and document type.

2. Time metrics calculation (Excel + Python)

Used Excel formulas to compute baseline metrics such as feedback time, revision time, and total closure time. Python extended the analysis into reviewer-specific latency, outlier detection, document-type patterns, and workload concentration.

3. Visualization dashboard

Built an Excel-based dashboard integrating pivot charts, conditional coloring, latency summaries, reviewer workload views, and overdue-item alerts. This created a shared visual layer for bottleneck recognition.

4. Workflow interpretation

Compared response speed, review load, and delay concentration across reviewers and document types to identify structural bottlenecks and opportunities for better coordination.

💻 Computational extension

from dashboard to observability

What Python adds

  • reviewer-specific mean and median turnaround time
  • latency variance by reviewer and document type
  • outlier detection for unusually delayed cases
  • workload concentration and imbalance scoring
  • trend analysis of response delay over time
  • bottleneck propagation patterns across review stages

Why it matters

Python makes the workflow argument more rigorous. Instead of only showing that review “felt slow,” it allows the project to identify where delay accumulates, which reviewers or stages generate more latency, and whether workload is distributed fairly across the system.

Possible deeper extensions

This project can be pushed further through rolling trend analysis, queue congestion modeling, document-type sensitivity analysis, and predictive flags for overdue or at-risk review items.

🧪 Workflow evaluation

coordination + transparency

What changed

The dashboard did not replace the validated system. Instead, it added an external observability layer that made timing, ownership, and workload visible through existing metadata.

manual updates shared signals better coordination

What was assessed

  • speed of bottleneck recognition
  • workload visibility across reviewers
  • confidence in decision-making
  • transparency of review ownership

📄 Project extension

prototype direction

This project is currently positioned as a workflow observability case study rather than a core paper in the portfolio. Its main contribution is to show how hidden review latency, workload imbalance, and bottleneck formation can be made visible through a compliance-safe coordination layer.

The broader design implication is that dashboarding in regulated environments is not just a reporting function. It can serve as a form of human-centered coordination support, helping teams interpret workflow conditions earlier without modifying validated systems.

💬 Reflection

why it is HCI

This project reframed transparency as a form of procedural equity. By transforming hidden review data into visible workflow signals, teams could move from intuition-driven coordination to evidence-based collaboration.

The key insight was that review delays are not only operational problems — they are also information-design problems. In this sense, the dashboard functioned not just as a reporting tool, but as a human-centered coordination layer for regulated work.
