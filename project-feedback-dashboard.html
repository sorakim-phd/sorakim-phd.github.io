Smart Feedback & Review Dashboard

workflow friction dashboard prototype TF-IDF clustering python streamlit deployed

Coordination breakdown is not sudden — it is produced, delay by delay, within review cycles. This prototype makes review delays, feedback loops, and revision burden visible without modifying validated documentation systems.

This project argues that review friction can be studied as an observability problem: delay, revision, and coordination become legible before they become formal failure.

Research Question

How can workflow friction be made visible in regulated systems without collapsing human judgment into automated outputs?

01

Problem

In regulated environments, document review is critical but opaque. Deviation, CAPA, and related records pass through multiple reviewers — yet visibility into feedback burden, timing, and revision-loop formation is limited.

Teams know a document is "still under review," but not what type of comments are driving the delay, where bottlenecks are forming, or whether a loop is escalating across rounds. The core problem is not delay itself — it is the lack of a shared operational view of that delay.

Observed pain points
  • Repeated revision loops difficult to observe as patterns
  • Feedback burden uneven across reviewers but hard to see
  • Small wording comments generated multi-day back-and-forth
  • Review process felt slow but source of friction remained unclear
HCI reframing

Review delay is an observability problem as much as a coordination problem. A missing visibility layer meant that friction accumulated without ever becoming legible enough to act on early.

02

Method

01
Dataset construction

Constructed a synthetic review-workflow dataset (100 records) representing comment categories, review rounds, severity, turnaround time, reopening, reviewer role, and document type.

02
UI prototype — 6-tab dashboard

Built an interactive dashboard in Python/Streamlit: workflow summaries, revision heatmaps, checklist assistant, rule-based comment classifier, document journey tracing, and reviewer simulation.

03
Visibility layer — TF-IDF clustering

Added TF-IDF similarity clustering for recurring issue detection, multi-factor revision burden scoring, and pattern summaries — as observability tools, not decision-automation systems.

04
Usability study · N=20

Prototype feedback session with QA/QC-experienced participants to assess clarity, interpretability, interface organization, and perceived usefulness.

04

Visibility Layer

The visibility layer adds computational pattern-detection on top of the rule-based dashboard. TF-IDF similarity clustering detects recurring issue patterns across review rounds — surfacing structural friction invisible in individual comment threads.

Same design principle as the UI prototype: makes hidden patterns legible without collapsing reviewer discretion into algorithmic output.

Open Visibility Layer ↗
Visibility layer · review friction explorer

TF-IDF clustering auto-detects repeated issue clusters. Pattern summary describes the selected document's review state in plain language.

What it does
  • TF-IDF clustering — auto-detects recurring patterns without manual tagging
  • Revision burden scoring — quantifies coordination load per document
  • Pattern summary — plain-language description of review state and bottleneck stage
What it does not do
  • Does not recommend actions or assign blame
  • Does not replace reviewer judgment
  • Does not modify validated documentation systems
  • Surfaces patterns for human interpretation only
Visibility is not a neutral interface property — it helps determine what becomes discussable, interpretable, and eventually accountable within review work.

Results showed relatively low variance across participants, which may reflect genuine agreement but also suggests the possibility of response bias in a small, conceptually aligned sample.

The main adoption barrier was conceptual, not interactional — the interface was clear, but the idea that review friction deserves structured visibility was unfamiliar to most participants.
06

What This Revealed

The main contribution is not efficiency — the prototype surfaces hidden review work as interpretable workflow signals, providing initial evidence that friction visibility is achievable without automating reviewer judgment.

More importantly, the study suggests that workflow friction is not merely a coordination inefficiency, but a structurally hidden form of labor that remains unaddressed because it is rarely made collectively visible. In that sense, the interface does more than summarize delay: it externalizes a class of review work that usually remains distributed, tacit, and difficult to discuss.

Participants did not struggle with the interface itself. The workflow-friction concept initially felt unfamiliar — pointing toward adoption barriers being conceptual rather than interactional. The next design challenge is therefore not interface clarity alone, but making the value of friction visibility legible to users trained to absorb review load silently.

Limitations
  • Synthetic dataset — findings require validation with real review records
  • Some participants initially struggled to interpret clustering outputs without guidance
  • Workflow-friction concept was conceptually unfamiliar, requiring onboarding
Novelty
  • External layer — operates without modifying validated systems
  • Non-automation principle — surfaces patterns, preserves human discretion
  • Responsibility-preserving design in compliance-constrained environments

07

Paper

Claim

Review friction in regulated environments is structural and largely invisible. This project provides initial evidence that a lightweight observability layer can surface it without modifying validated systems or replacing reviewer judgment.

What it does
Provides initial evidence that review friction, comment burden, and revision loops can be surfaced through a computational support prototype — and that AI can serve as a pattern-detection layer rather than a decision-automation system.
