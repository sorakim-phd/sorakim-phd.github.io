🧩 Background

navigation as a research site

GMP procedural documentation is not a flat archive — it is a network. A single deviation investigation may require navigating across SOPs, method validation documents, specification references, and training records before a root cause can be identified. These documents cross-reference each other, but the links are textual rather than interactive, and the navigation burden falls entirely on the practitioner.

This creates a research opportunity: if we can observe how practitioners actually move through documents — where they go first, where they return, where they hesitate — we can surface structural properties of the documentation system that are otherwise invisible. Navigation behavior is a behavioral trace of documentation friction.

💬 Where a practitioner loops back is not a mistake. It is a signal that the document structure failed to resolve a judgment demand the first time.

⚠️ Observation

navigation friction in practice

Observed pain points

  • Cross-references in GMP documents are textual ("refer to SOP-QC-004") rather than navigable, placing linking work on the practitioner.
  • Related documents are distributed across the archive without clear pathways between them.
  • Practitioners rely heavily on prior experience to know where to look — less experienced users face significantly higher navigation burden.
  • Loopback behavior (returning to a document already visited) is common but invisible in standard documentation workflows.

Why this is worth tracking

Navigation behavior contains information that practitioners cannot always articulate. A loopback reveals that something in the previously visited document was not resolved — but without tracking, that signal disappears. This prototype makes those signals visible, turning implicit navigation friction into observable, structured data.

🎯 Design question

HCI framing

Can a lightweight document navigation prototype make practitioner movement through GMP procedural archives observable — and can the resulting behavioral data reveal structural properties of the documentation system that are otherwise hidden?

🧠 Method

interactive prototype + behavioral tracking

1. Document set design

Constructed a set of 7 synthetic GMP documents (SOPs, analytical method protocol, reference specification) designed to reflect a realistic OOS investigation workflow — with intentional cross-referencing fragmentation to create naturalistic navigation demands.

2. Scenario

Participants were given a standardised OOS scenario (ELISA assay result exceeding specification) and asked to find the relevant procedures and documents. No guidance or hints were provided.

3. Behavioral tracking

The prototype automatically recorded from_doc → to_doc transitions, action type (forward / cross-reference / loopback), visit count per document, and time spent per document. Data was exported as CSV per participant session.

4. Usability study

Conducted with 5 GMP-experienced practitioners. Post-task survey measured navigation clarity, cross-reference usefulness, real-world utility, and collected qualitative feedback on confusion points and improvement suggestions.

🖥 Prototype

interactive document navigator

The Cognitive Navigation Explorer presents participants with a GMP document set and an OOS investigation scenario. As participants navigate between documents, their movement is tracked in real time. At the end of the session, a results screen visualises the navigation path as a directed network graph — showing forward moves, cross-reference jumps, and loopbacks.

🧪 Usability study

N = 5 · GMP-experienced · OOS investigation background

🔑 Key finding: loopbacks were universal — and their source was the document structure, not the tool

All five participants revisited at least one document during their session. The most common loopback pattern was: Analyst Training → ELISA Test Method SOP → Method Validation → ELISA Test Method SOP → OOS Procedure. When asked where they felt confused, participants with less experience cited difficulty understanding which document to link to next — and explicitly noted that this was a property of the SOP structure, not the prototype.

First document opened

4/5 participants opened an analyst-related document first. Their reasoning: pipetting errors are the most common cause of ELISA OOS results in practice — reflecting experience-based heuristics rather than document-structure guidance. 1/5 opened the Reference Specification first to establish the numeric acceptance criteria before investigating.

Experience vs. navigation difficulty

The 3 participants with 5+ years of GMP experience reported no confusion about navigation direction. The 2 participants with approximately 1 year of experience reported difficulty identifying which document to navigate to next via cross-references — but attributed this to SOP structure ambiguity, not to the prototype interface.

The key insight

When less experienced practitioners struggled with cross-reference navigation, they did not blame the tool — they identified the SOP itself as the source of ambiguity. This is exactly the argument that this research program has been building: navigation difficulty in GMP documentation is not a user competence problem. It is a documentation design problem. The prototype made that distinction visible by separating interface usability (rated 5.0/5) from structural document friction (identified as the actual source of confusion).

💬 Qualitative feedback

The most consistent improvement request was for a more comprehensive document set that better reflects the full scope of real GMP investigations. This is an expected limitation of a synthetic prototype — and a clear direction for future development. No participants identified the interface itself as a source of friction, which confirms that the tool successfully isolated document-structural friction from interface-level usability issues.

🔗 Connection to the broader research program

navigation → friction → accountability

This project sits between the Cross-Reference system (which maps document-level linkage gaps computationally) and the Premature Causal Convergence study (which shows what happens when investigators are given time to reason freely). Where the Cross-Reference system asks "what connections are missing in the document archive?", this project asks "where do practitioners actually get stuck when they try to follow those connections?"

The finding that less experienced practitioners struggled with cross-reference navigation — and that they correctly identified the SOP structure, not the interface, as the source of that difficulty — is a behavioral confirmation of the core research argument: documentation design shapes cognitive navigation, and cognitive navigation shapes investigation quality.

document structure → navigation behavior → loopback / hesitation → investigation friction

📄 Paper in progress

research output

This project is being developed into a paper on behavioral navigation tracking as a method for surfacing documentation friction in regulated environments. The paper contributes a methodological argument: navigation behavior is a valid and tractable proxy for document-structural friction — and lightweight interactive prototypes can make that behavior observable without requiring deployment inside validated systems.

💬 Reflection

why it matters

The most important finding from this study was not a usability score — it was a qualitative distinction. When less experienced participants struggled with cross-reference navigation, they did not say "this tool is confusing." They said "I can't tell where to go because the SOP doesn't make it clear." That is the research finding: the interface was legible, but the underlying document structure was not.

This distinction — between interface friction and structural friction — is what the broader research program is trying to make visible. The Cognitive Navigation Explorer is a tool for producing that distinction empirically, in a setting that practitioners recognize as realistic.
