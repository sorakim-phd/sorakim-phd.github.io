🧩 Background

mental workflows

In regulated GMP environments, analysts and QA reviewers are expected to follow highly structured Standard Operating Procedures (SOPs). On paper, these workflows appear linear and orderly.

In practice, however, human reasoning is rarely linear. People pause, revisit earlier steps, reinterpret instructions, compare alternatives, and reconstruct meaning in context. This creates a hidden gap between procedural logic — how the workflow is formally written — and cognitive logic — how the workflow is actually navigated in real use.

That mismatch matters because when procedural structure and human reasoning diverge, users experience more confusion, more re-checking behavior, and higher cognitive burden in high-stakes documentation work.

⚠️ Observation

hidden friction in thinking

Observed cognitive frictions

  • Users moved back and forth between sections rather than following written order step-by-step.
  • Repeated verification occurred around terminology, prior records, and ambiguous instructions.
  • Interruptions made it difficult to resume from the same reasoning state.
  • Users often had to interpret what a step meant in context rather than simply execute it.

Why this mattered

These behaviors were not signs of carelessness. They reflected a structural issue: procedural systems often assume linear compliance, while human reasoning works through loops, detours, comparison, and reconstruction.

💬 Procedural systems are usually written as straight lines, but human reasoning often unfolds as a path with returns, branches, pauses, and repairs.

🎯 Design question

HCI framing

How can we capture and visualize human cognitive pathways within validated workflows to reveal where procedural rigidity conflicts with natural reasoning flow?

🧠 Method

observation + interviews + modeling

1. Field observation

Observed analysts and QA reviewers during deviation and CAPA documentation work. Recorded hesitation points, repeated look-backs, interruption moments, and step reversals during procedural tasks.

2. Cognitive interviews

Conducted reflective think-aloud and retrospective questioning, such as “What were you thinking here?” and “Why did you return to that step?” These interviews helped reconstruct reasoning sequences beyond what was visible on the surface.

3. Cognitive mapping (Miro / Lucidchart)

Translated observed reasoning sequences into cognitive flow diagrams. Nodes represented decision points, arrows represented mental transitions, and repeated returns were marked as cognitive bottlenecks.

4. Quantification (Excel + Python)

Used Excel to log pauses, detours, loop-backs, and interruption points. Then used Python to convert reasoning paths into transition matrices, calculate loop frequency, and visualize cognitive flow as analyzable network structures.

💻 Computational extension

making thought flow measurable

What Python adds

  • transition matrix construction for reasoning paths
  • loop frequency and hesitation density analysis
  • comparison of procedural order vs. cognitive traversal order
  • high-centrality bottleneck node detection using networkx
  • visualization of reasoning flow as network graphs

Why it matters

Python makes the cognitive argument more rigorous. Instead of only saying that users felt confused, it allows the project to show where reasoning repeatedly returns, which steps attract cognitive traffic, and how strongly written order differs from actual mental navigation.

Possible deeper extensions

This project can be pushed further through transition-probability modeling, bottleneck centrality analysis, mismatch scoring between written SOP order and cognitive order, hesitation clustering by section, and task-time coupling between loop density and completion time.

🧪 Evaluation logic

qualitative + computational evidence

What was compared

The project compared the formal procedural path written in SOP-like structures with the actual cognitive path reconstructed from observation and interviews.

written order actual path mismatch

What was assessed

  • loop-back frequency
  • hesitation density
  • interruption sensitivity
  • bottleneck concentration by step
  • relationship between cognitive friction and task time

Why the result matters

Rather than showing that users “failed” to follow procedures, the project showed that procedures themselves distribute uneven cognitive demands. The problem was not only user difficulty, but structural mismatch.

📄 Project extension

prototype direction

This project is currently positioned as a cognitive systems case study rather than a core paper in the portfolio. Its main contribution is to show that reasoning paths in regulated work can be observed, reconstructed, and modeled as structured workflow behavior rather than treated as invisible individual variation.

The broader design implication is that the gap between written procedure and actual cognitive navigation can be made visible and analyzable. In this sense, the project functions as a bridge between procedural design, cognitive friction, and human-centered systems research.

💬 Reflection

why it is HCI

This project revealed a structural mismatch between how procedures are written and how humans think. By combining qualitative observation with computational path analysis, it became possible to transform invisible reasoning patterns into design-relevant evidence.

The point is not to make regulated systems less rigorous. It is to make them more cognitively realistic. Procedural equity means recognizing that compliance is not only about whether a step exists, but also about whether a human can navigate that step without unnecessary cognitive strain.
