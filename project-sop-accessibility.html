SOP Accessibility System

Procedural systems do not only store information — they structure how interpretation unfolds.

document UX cognitive friction python streamlit AI support layer

Access alone does not resolve interpretive burden in procedural systems. In pharmaceutical manufacturing, Standard Operating Procedures (SOPs) define how every critical task is performed — and when those documents are hard to find, friction travels downstream into hesitation, workarounds, and judgment under constraint. This project builds a compliance-safe retrieval layer that treats document accessibility as an HCI problem.

// 01

Problem

Procedural archives are often stored as scanned PDFs — blocking keyword search, burying relevant sections behind similar-sounding titles, forcing users to rely on folder memory. The problem isn't missing functionality. It's a structural mismatch between how archives are organized and how people actually search under constraint.

// pain points
  • Scanned files blocked keyword search
  • Titles too broad, similar, or generic
  • Users opened multiple wrong files
  • Access required archive naming familiarity
  • Time pressure amplified every friction point
// implication

Accessibility problems systematically introduce hesitation and interpretive variance before any formal task error occurs. What looks like individual failure downstream often begins as structural difficulty upstream.

// 02

Method

01
metadata & keyword mapping

Reviewed approximately 500 procedural documents. Extracted keywords, document identifiers, page references, and process categories to build a searchable metadata index.

02
structure & overlap analysis

Used Python to inspect overlapping terms, recurring categories, and keyword patterns — identifying where users face ambiguity from similar titles and mismatched labels.

03
prototype development

Built an external retrieval layer allowing keyword-based search without altering validated source files. Compliance-safe by design.

04
usability study · N=12, think-aloud protocol

Used the prototype as both a design intervention and an analytic probe — observing how documentation access shapes confidence, search behavior, and navigation experience.

// 04

AI Support Layer

Retrieval is only one component of procedural usability. Rather than optimizing access alone, this layer makes visible why results appear, what adjacent procedures matter, and where ambiguity remains — maintaining reasoning openness rather than collapsing it prematurely. It does not replace reading or make compliance decisions.

Open AI Layer ↗
AI support layer · demo video

Three design principles made explicit upfront: EXPLAINS why a result appeared, SURFACES related procedures, KEEPS interpretive options open.

This prototype suggests that retrieval is only one component of procedural usability, and that interpretive support is necessary to maintain reasoning openness in regulated environments.

// what worked
  • Task-relevant keyword entry (not filename memory)
  • Snippets reduced early-stage uncertainty
  • Document IDs improved search confidence
// remaining friction
  • No-result states still ambiguous (wording vs. coverage?)
  • Similar titles persisted as differentiation problem
  • Points to gaps in metadata design, not just interface
The strongest signal: ambiguity became more visible. The prototype helped separate user difficulty from structural difficulty in the archive itself.
// 06

What This Revealed

Document accessibility shapes how responsibility is interpreted in practice. Retrieval difficulty shapes what people notice, how they hesitate, and which document they trust — and when those patterns appear downstream as inconsistency or deviation, they are misread as individual failure rather than structural friction.

retrieval difficulty → interpretive variance → judgment under constraint → perceived responsibility

Future Work

// next steps
  • Context-sensitive retrieval explanation (why this result, not just what)
  • Adjacent procedure recommendation from cross-reference mappings
  • Ambiguity logging across repeated searches to surface structural gaps
// design constraint

All extensions must preserve human interpretive authority — no black-box decisions, no modification of validated source files.

// 08

Paper in Progress

This paper examines how accessibility constraints in SOP systems shape interpretive variability and downstream responsibility judgments. The central argument: accessibility improvements in validation-sensitive systems can be achieved through external support layers — without touching validated source files.
