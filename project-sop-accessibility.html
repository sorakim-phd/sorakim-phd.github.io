🧩 Background

problem framing

In regulated and procedure-heavy environments, formal operating documents shape how work is performed, checked, and reviewed. These documents are not merely references; they are part of the infrastructure through which high-stakes work becomes legible and repeatable.

In practice, however, procedural archives are not always easy to use. Digital versions are often stored as scanned PDFs, which makes them difficult to search, cross-reference, or navigate efficiently. As a result, users may rely on file titles, memory, and manual scrolling simply to find the right procedure.

This burden is not distributed evenly. In document-heavy systems, more experienced workers often accumulate informal knowledge of naming conventions, archive structure, and likely file locations, while less experienced workers may take much longer to locate the same material. In that sense, access to procedural information can become partly experience-dependent, even when the formal system is the same for everyone.

💬 Accessibility is often invisible in regulated systems, yet it strongly shapes efficiency, confidence, and cognitive burden in everyday work.

⚠️ Problem

document friction

Observed pain points

  • Scanned procedural files blocked keyword search.
  • Document titles were often too broad or too similar.
  • Users opened multiple wrong files before locating the correct one.
  • Less experienced users faced higher retrieval burden because navigation often depended on prior familiarity with the archive.
  • Time-sensitive review situations amplified navigation burden and cognitive fatigue.

Why this was difficult

In controlled environments, source documents cannot be casually rewritten or transformed into a new interface. That meant the design challenge was not “replace the system,” but rather “improve accessibility without touching the archive itself.”

🎯 Design question

HCI framing

How can a human-centered retrieval layer improve accessibility and reduce cognitive friction in a constrained environment where only approved materials can be used operationally?

↔ System change illustration

before / after

Before

  • Title-based browsing across scanned files
  • Keyword search unavailable or unreliable
  • Users manually opened multiple documents
  • Relevant sections had to be found by scrolling and memory
  • Similar titles increased uncertainty during retrieval

After

  • Keyword-based entry into the document archive
  • Ranked results with snippets and highlighted matches
  • Document identifiers and page cues surfaced early
  • No-result states handled more explicitly
  • A lightweight support layer improved navigation without changing source files
This project shows how a lightweight accessibility layer can improve procedural document navigation without altering controlled source documents.

🧠 Method

low-code + mixed methods

1. Metadata and keyword mapping

Reviewed approximately 500 procedural documents and extracted representative keywords, document identifiers, page references, and related process categories to build a searchable metadata index.

2. Structure and overlap analysis

Used Excel and Python to inspect overlapping terms, recurring categories, and retrieval-relevant keyword patterns across documents.

3. Prototype artifact

Created an external retrieval aid rather than a replacement system: a prototype accessibility layer that allowed users to search by keywords, document cues, and related terminology without altering validated source files.

4. Prototype-based usability testing

Conducted usability testing with likely end users using the prototype artifact. The artifact was not deployed as a live production tool; instead, it was used as a testable representation of a compliance-safe accessibility layer.

🧪 Usability testing

prototype evaluation

Testing setup

Because controlled environments cannot freely adopt unvalidated interface changes, the prototype was evaluated outside the live system. Participants were asked to complete simple retrieval tasks using the prototype as a simulated accessibility layer.

prototype artifact task-based test n = 12

What was measured

  • ease of finding relevant document sections
  • usefulness of titles, snippets, and highlighting
  • overall intuitiveness of the interface
  • qualitative feedback on confusion and ambiguity

Quantitative summary

A short usability survey was completed by 12 participants. Most respondents reported regular SOP use (91.7%) and daily SOP search behavior (91.7%). The prototype received high ratings for ease of use (mean 4.83/5), usefulness of search results (mean 4.50/5), and overall intuitiveness (mean 4.82/5).

regular SOP users: 91.7% daily search: 91.7% ease: 4.83 / 5 usefulness: 4.50 / 5 intuitiveness: 4.82 / 5

Why this matters

The study did not claim production deployment inside a validated system. Instead, it asked a more HCI-relevant question: if a compliance-safe accessibility layer were available, would actual users navigate procedural information more efficiently and with less uncertainty?

💬 Qualitative feedback

Participants were generally very positive about the prototype, but a few noted moments of confusion when a query returned no result or when multiple entries with similar titles appeared under different document IDs. Some of these issues were also related to the synthetic prototype corpus, which occasionally produced missing or overlapping entries. These observations pointed to future improvements in query feedback, document differentiation, and result explanation.

📄 Paper in progress

research output

This project is being developed into a paper on accessibility, document usability, and compliance-constrained design in procedural environments. The paper extends the case study from a practical prototype into a broader HCI argument: accessibility improvements in validation-sensitive systems can often be achieved through external support layers rather than direct modification or replacement of the underlying system.

The paper-in-progress uses this project to frame a broader question about how human-centered design can operate in environments where formal systems are difficult to modify, yet users still experience substantial navigational and cognitive burden.

💬 Reflection

why it is HCI

This project showed that accessibility problems in constrained procedural environments are not trivial inconveniences; they shape time, confidence, navigation, and cognitive load in everyday work. In other words, document control may preserve compliance, but it can also generate hidden interaction costs.

From an HCI perspective, the important insight was that meaningful design improvement did not require rewriting the validated system. An external, low-code accessibility layer was enough to improve user performance and reveal a concrete design opportunity.

Importantly, the study also showed that accessibility is not only about successful retrieval. Users needed clearer feedback when no result was returned and better differentiation between similarly titled entries. These observations helped frame the project not simply as a search aid, but as a broader HCI problem concerning uncertainty, navigation, and procedural interaction in regulated work environments.
