🧩 Background

document overload

In Good Manufacturing Practice (GMP) environments, every Deviation, CAPA, and Change Control record must be cross-linked to demonstrate process integrity and inspection readiness. These records are often deeply interdependent, but they usually exist as separate validated Word files.

Because these files cannot be freely restructured or automatically linked, analysts and QA reviewers must repeatedly copy and verify identifiers such as deviation IDs, batch numbers, product codes, status labels, and closure dates across multiple records.

What looks like a traceability requirement on paper becomes, in practice, a cognitive burden placed on people.

⚠️ Observation

field-level inefficiencies

Observed pain points

  • Users relied on memory or keyword search to link related deviations and CAPAs.
  • Inconsistent file naming increased the risk of outdated or incorrect version use.
  • QA reviewers spent excessive time checking document status and closure consistency.
  • Manual linkage work created fatigue during audits and inspection preparation.

Why this mattered

These were not individual user mistakes. They were consequences of system design under validation constraints, where automation was limited and humans became the connectors of information across fragmented records.

💬 In validation-constrained systems, people often become the missing infrastructure between documents that cannot easily talk to each other.

🎯 Design question

HCI framing

How can document traceability be enhanced and cognitive load reduced without breaching validation requirements or introducing new software systems?

🧠 Method

metadata mapping + workflow analysis

1. Excel-based cross-reference sheet

Extracted key metadata fields such as Deviation ID, CAPA ID, batch number, product code, status, and closure date. Constructed an interactive Excel matrix linking related records and added status-based filters such as open, under review, and closed.

2. Keyword indexing layer

Developed a thematic index summarizing deviation root cause categories such as equipment, documentation, and operator-related factors. This enabled rapid filtering across records and helped detect recurring patterns.

3. Workflow evaluation

Compared average search time before and after the intervention, while also collecting qualitative feedback on clarity, efficiency, and perceived workload reduction.

4. Computational verification

Used Python to automate metadata cleaning, linkage checking, and traceability validation across records. This extended the project from manual cross-reference support into computationally testable document structure analysis.

💻 Computational extension

python depth

What Python adds

  • automated cross-link checks across hundreds of records
  • identifier-matching verification using pandas
  • version inconsistency and missing-link detection
  • network-style analysis of how deviations, CAPAs, and changes connect
  • visual evidence through linkage heatmaps and structure summaries

Why it matters

Python gives analytical rigor to the traceability argument. Instead of only showing that the cross-reference system was convenient, it allows the project to demonstrate that record linkage itself has structure — and that weak linkage points can be identified, measured, and improved.

Network thinking

This project can also be framed as a lightweight document-network analysis: not just whether records exist, but how they connect, where they cluster, and where traceability breaks down.

🧪 Evaluation

workflow impact

Evaluation setup

Because the validated system itself could not be modified, the intervention remained external to the live controlled environment. The cross-reference matrix functioned as a human-readable support layer rather than a replacement platform.

metadata layer faster lookup lower burden

What was assessed

  • document tracking speed
  • version consistency
  • clarity of process overview
  • perceived workload reduction
  • user confidence in traceability checks

📄 Paper in progress

research output

This project is being developed into a paper on traceability, metadata infrastructure, and cognitive load in compliance-constrained documentation systems. The paper extends the cross-reference intervention into a broader HCI argument about how validation constraints shift linking work onto users.

In paper form, the project contributes to research on regulated knowledge systems by showing that traceability is not only a technical requirement but also an interaction and cognition problem shaped by documentation architecture.

💬 Reflection

why it is HCI

This project demonstrated that compliance and usability are not mutually exclusive. Even within the constraints of validated documentation, human-centered design can restore clarity, efficiency, and cognitive equity.

Rather than replacing existing systems, redesigning how people interact with them can transform documentation from a bureaucratic burden into a cognitive support structure that sustains both accuracy and well-being.
